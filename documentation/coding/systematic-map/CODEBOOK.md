## What this document is

This is the complete, standalone guide for **D5.6 — systematic map data extraction**. It covers:

- What you are coding and why
- The coding schema — every field, every valid value, with examples
- How each coding round works step by step
- How to record your codes
- How to reconcile disagreements
- How to interpret inter-rater reliability (IRR) statistics
- Stopping rules — when the codebook is ready for the full corpus run

Both **human coders** and the **LLM** use the same coding schema. The only difference is that humans code by reading the paper; the LLM receives the text and the schema as a structured prompt.

---

## Background and context

This systematic map examines how climate adaptation **processes and outcomes** are tracked and measured for smallholder agricultural producers in low- and middle-income countries (LMICs). It follows CEE/Campbell Collaboration standards.

The corpus contains **~6,200 records** that passed abstract screening. Around 4,000 full texts have been retrieved. Data extraction codes all included records against a **19-field schema** (protocol Table 3, Deliverable 3) covering study characteristics, producer types, adaptation domains, indicators, and methods.

Coding is done **iteratively in rounds of 100 papers.** In each round, two human coders and the LLM code independently. After reconciliation, the codebook and LLM guidance are updated before the next round. The LLM runs the full corpus only once the codebook is locked.

## Overview of each round

Each round follows the same six steps:

```
Step 1 — Code independently
         H1, H2, and LLM all code the same 100 papers independently and simultaneously.
         No discussion between coders during this step.
         ↓
Step 2 — Reconcile (H1 + H2 only)
         H1 and H2 compare their codes field by field, discuss disagreements,
         and agree on a single gold-standard decision for every record.
         ↓
Step 3 — Compute IRR
         Run the IRR script to compare: H1 vs H2, LLM vs gold standard.
         Kappa and sensitivity computed per field and overall.
         ↓
Step 4 — Review disagreements
         Identify fields with low kappa. Review the papers that caused disagreement.
         Ask: was the coding guidance unclear? Were the categories insufficient?
         ↓
Step 5 — Update codebook and LLM criteria
         Update codebook_vN.md with clarified definitions and new examples.
         Update CODEBOOK.md with clarified definitions and new examples.
         Copy llm-criteria/criteria_vN.yml → criteria_v(N+1).yml with field-level
         improvements. Log the change in llm-criteria/CHANGELOG.md.
         ↓
Step 6 — Check stopping rules
         If all closed fields meet κ ≥ 0.60 AND sensitivity ≥ 0.80 AND
         emergent fields are saturated → lock codebook, run full corpus.
         Otherwise → proceed to next batch.
```

## The coding schema (19 fields)

Use this reference when coding every paper. Each field entry includes:
- **What it captures**
- **Valid values** — use these exactly (except emergent fields, which are free text)
- **Instructions and examples**

---

### 1. `publication_year`
**What:** Year the study was published.
**Valid values:** Integer, e.g. `2021`
**Instructions:** Use the most recent version date if the paper appears in multiple versions (preprint + journal). If only a range is given, use the end year.

---

### 2. `publication_type`
**What:** Type of publication.
**Valid values:** `journal_article` | `report` | `working_paper` | `thesis` | `other`
**Instructions:**
- `journal_article` — peer-reviewed journal (including open access journals)
- `report` — project or programme report, policy brief, evaluation report
- `working_paper` — pre-publication working paper, discussion paper
- `thesis` — PhD or master's dissertation
- `other` — book chapter, conference paper, dataset paper

---

### 3. `country_region`
**What:** Country or countries where the study was conducted.
**Valid values:** Free text. Separate multiple countries with semicolons.
**Instructions:**
- If specific countries are named, list them: `Kenya; Ethiopia`
- If only a region is stated (no countries named): `Sub-Saharan Africa` or `South Asia`
- Multi-country studies covering a whole region: list the region, not all countries
- If the study is a global review with no specific country focus: `Global`

**Examples:**
- `Tanzania` — single country
- `Ghana; Mali; Burkina Faso` — three countries
- `East Africa` — regional, countries not specified

---

### 4. `geographic_scale`
**What:** Spatial scale of the study's analysis.
**Valid values:** `local` | `sub-national` | `national` | `multi-country` | `regional`
**Instructions:**
- `local` — village, community, watershed, single site
- `sub-national` — district, province, state, region within a country
- `national` — one country, national-level analysis
- `multi-country` — two or more specific countries
- `regional` — broad region (e.g. West Africa, South Asia) without country-level focus

---

### 5. `producer_type`
**What:** Type(s) of agricultural producers studied.
**Valid values:** `crop` | `livestock` | `fisheries_aquaculture` | `agroforestry` | `mixed`
**Instructions:**
- Select all that apply, separated by semicolons: `crop; livestock`
- `mixed` — explicitly multi-system (e.g. mixed crop-livestock), or "smallholder farmers" with no further specification
- If the paper studies a specific crop (e.g. rice farmers) → `crop`
- Pastoralists → `livestock`
- Artisanal fishing communities → `fisheries_aquaculture`

---

### 6. `marginalized_subpopulations`
**What:** Whether the study explicitly targets or analyses any marginalized groups.
**Valid values:** `women` | `youth` | `landless` | `indigenous_peoples` | `ethnic_minorities` | `migrant_seasonal_workers` | `none_reported`
**Instructions:**
- Select all that apply, separated by semicolons
- Code only what is explicitly stated in the paper — do not infer
- A study that disaggregates results by gender but does not specifically target women → still code `women` if gender is explicitly discussed
- If no marginalized groups are mentioned → `none_reported`

---

### 7. `adaptation_focus` ⚠ *Emergent field — free text*
**What:** The specific climate adaptation action, intervention, or practice that the study tracks or evaluates.
**Valid values:** Free text — describe in one concise phrase.
**Instructions:**
- This field is emergent: capture what the study is actually about, not a generic category
- Be specific enough to be useful, brief enough to scan: aim for 5–15 words
- Include both the adaptation type AND the domain if relevant

**Examples:**
- `drought-tolerant maize varieties`
- `seasonal climate forecasts for smallholder decision-making`
- `index-based livestock insurance`
- `community-based watershed management`
- `farmer field schools for climate-resilient agriculture`
- `crop diversification as risk management strategy`

**Codebook note:** After Round 1, tally the free-text codes and group into emerging themes. These themes become the basis for a controlled vocabulary in later rounds.

---

### 8. `domain_type`
**What:** Whether the study assesses adaptation processes, outcomes, or both.
**Valid values:** `adaptation_process` | `adaptation_outcome` | `both`
**Instructions:**
- **Process** — studies that measure *whether/how* farmers adopt, learn, participate, or engage with adaptation (the steps toward adaptation)
- **Outcome** — studies that measure *what happened* as a result of adaptation (yields, income, wellbeing, resilience)
- **Both** — study tracks both the process (e.g. adoption of a practice) and its outcome (e.g. yield impact)
- When in doubt: a study measuring crop yield changes = outcome; a study measuring awareness or adoption rates = process

---

### 9. `process_outcome_domains`
**What:** The specific process and/or outcome domains assessed.
**Valid values:** Select all that apply, separated by semicolons.

**Process domains:**
| Code | Meaning |
|------|---------|
| `knowledge_awareness_learning` | Awareness of climate change, knowledge of adaptation options, learning outcomes |
| `decision_making_planning` | Farmer decision-making, planning processes, uptake of information in decisions |
| `uptake_adoption` | Rates of adoption of a practice or technology |
| `behavioral_change` | Changes in farming behaviour, land management practices |
| `participation_coproduction` | Participation in programmes, co-design, farmer group engagement |
| `institutional_governance` | Policy uptake, institutional change, governance arrangements |
| `access_information_services` | Access to extension, credit, markets, climate information services |

**Outcome domains:**
| Code | Meaning |
|------|---------|
| `yields_productivity` | Crop yield, livestock productivity, fish catch, productivity metrics |
| `income_assets` | Household income, asset accumulation, financial outcomes |
| `livelihoods` | Broader livelihood outcomes, food security, diversification |
| `wellbeing` | Health, nutrition, subjective wellbeing |
| `risk_reduction` | Reduced exposure or sensitivity to climate shocks |
| `resilience_adaptive_capacity` | Adaptive capacity, resilience indices, vulnerability indices |

**Examples:**
- A study measuring adoption rates of drought-tolerant seeds: `uptake_adoption`
- A study measuring yield and income impacts of an irrigation scheme: `yields_productivity; income_assets`
- A study on participatory scenario planning and farmer learning: `knowledge_awareness_learning; participation_coproduction`

---

### 10. `indicators_measured` ⚠ *Emergent field — free text*
**What:** The specific indicators or metrics used in the study.
**Valid values:** Free text — list the key indicators, comma-separated.
**Instructions:**
- Extract actual indicators mentioned in the methods/results, not general statements
- Include units where given: `yield (kg/ha)`, `income (USD/season)`
- For indices: name the index: `self-reported adaptive capacity index (5-point scale)`

**Examples:**
- `yield (t/ha), household food security score (HFIAS), income diversification index`
- `adoption rate (%), area under improved variety (ha)`
- `seasonal temperature anomaly, crop failure probability, coping strategy index`

---

### 11. `methodological_approach`
**What:** Primary methodological design.
**Valid values:** `qualitative` | `quantitative` | `mixed_methods` | `participatory` | `modeling_with_empirical_validation`
**Instructions:**
- `qualitative` — interviews, focus groups, ethnography, document analysis only
- `quantitative` — surveys, trials, observational data, statistical analysis
- `mixed_methods` — explicit combination of both qualitative and quantitative
- `participatory` — participatory rural appraisal, participatory mapping, photovoice, farmer-led trials — use when participation is the *primary* methodology
- `modeling_with_empirical_validation` — crop models, climate models, agent-based models with field data used for validation

---

### 12. `purpose_of_assessment`
**What:** Primary purpose of the measurement/tracking exercise as stated or implied.
**Valid values:** `project_learning` | `program_evaluation` | `donor_reporting` | `national_reporting` | `research`
**Instructions:**
- `research` — academic study with no specific programme affiliation
- `program_evaluation` — commissioned evaluation of a specific programme or project
- `project_learning` — monitoring and learning embedded within an ongoing project
- `donor_reporting` — explicitly for accountability to a funder
- `national_reporting` — contributing to NAPs, NDCs, national monitoring frameworks

---

### 13. `data_sources`
**What:** Data sources used.
**Valid values:** Select all that apply, separated by semicolons.
- `surveys` — household surveys, farm-level questionnaires
- `administrative_data` — government records, market data, satellite-derived crop statistics
- `remote_sensing` — satellite imagery, drone data
- `participatory_methods` — PRA, FGDs, key informant interviews
- `secondary_data` — existing datasets, meta-analysis, systematic review data

---

### 14. `temporal_coverage`
**What:** Time horizon of data collection.
**Valid values:** `cross_sectional` | `seasonal` | `longitudinal`
**Instructions:**
- `cross_sectional` — single point in time or single season
- `seasonal` — tracks across one or a few seasons (intra-annual)
- `longitudinal` — multi-year data, panel data, repeated measures across years

---

### 15. `cost_data_reported`
**What:** Whether cost or resource requirement data are reported (e.g. cost-effectiveness, cost of implementation).
**Valid values:** `yes` | `no`
**Instructions:** Code `yes` only if the paper explicitly reports cost figures, cost-effectiveness ratios, or resource requirements for the adaptation. Mention of "limited resources" without numbers = `no`.

---

### 16. `equity_inclusion`
**What:** Whether equity dimensions are explicitly addressed.
**Valid values:** Select all that apply, separated by semicolons. Use `none_reported` if none.
- `gender` — gender-disaggregated analysis or gender-targeted interventions
- `youth` — explicit focus on young farmers or age-disaggregated outcomes
- `land_tenure` — land access or tenure security as a variable
- `disability` — explicit inclusion of persons with disabilities
- `other` — any other equity dimension stated (e.g. caste, religion)
- `none_reported` — equity not mentioned

---

### 17. `strengths_and_limitations`
**What:** Author-reported strengths and limitations of the study, combined in one field. This is a single free-text field per protocol Table 3 ("Strengths and limitations — Author-reported strengths and limitations — extract verbatim where possible").
**Valid values:** Free text.
**Instructions:**
- Look in the Discussion, Conclusion, and Limitations sections
- Label each component: `Strength: ...` / `Limitation: ...`
- Extract verbatim where authors use evaluative language; paraphrase briefly if they do not have an explicit section
- If only limitations are stated (no strengths), record just the limitations with the `Limitation:` prefix

**Example:**
> `Strength: large sample across three agroecological zones. Limitation: single-season follow-up does not capture inter-annual variability. Limitation: self-reported yield data may be subject to recall bias.`

---

### 18. `lessons_learned`
**What:** Key lessons or recommendations reported by the authors.
**Valid values:** Free text. Summarise in 1–3 sentences.
**Instructions:** Focus on what the authors say should be done differently or what they recommend for policy/practice. Not the same as findings.

---

### 19. `validity_notes`
**What:** Notes relevant to internal or external validity.
**Valid values:** Free text.
**Instructions:** Flag issues that affect how much weight this study can carry:
- Small sample (n < 30)
- Single site (limits generalisability)
- Self-reported outcomes (social desirability bias risk)
- Short follow-up (< 1 season)
- No control or comparison group
- Attrition not reported

## Instructions for human coders

### Before you start
1. Read this entire guide before coding your first paper.
2. Re-read the schema section of this CODEBOOK — it is updated after each round with new clarifications and examples.
3. Do not discuss your codes with the other coder until Step 2 (reconciliation).

### How to code each paper
1. Open the PDF/HTML from `scripts/results/FT-R{round}{batch}/`
2. Open your coding sheet: `documentation/coding/systematic-map/rounds/FT-R{round}{batch}/coding_H1.csv` (or H2)
3. For each paper (one row), fill in all 19 fields using the schema above
4. If a field is genuinely not reported or not applicable: leave blank or enter `not_reported`
5. If you are unsure: make your best judgement and add a note in `validity_notes`

### Your coding sheet format

The CSV has these columns (in order):

```
doi, filename, publication_year, publication_type, country_region,
geographic_scale, producer_type, marginalized_subpopulations,
adaptation_focus, domain_type, process_outcome_domains,
indicators_measured, methodological_approach, purpose_of_assessment,
data_sources, temporal_coverage, cost_data_reported, equity_inclusion,
strengths_and_limitations, lessons_learned, validity_notes, coder_id, notes
```

The first two columns (`doi`, `filename`) are pre-filled. You fill the rest.

---

## Instructions for reconciliation (Step 2)

Reconciliation is done by H1 and H2 together after both have coded independently.

1. Open both coding sheets side by side
2. For each paper and each field:
   - If H1 and H2 agree → enter that value in `reconciled.csv`
   - If H1 and H2 disagree → discuss until you reach agreement, then enter the agreed value
3. Record the reason for any substantive disagreement in the `reconciliation_notes` column of `reconciled.csv` — this feeds directly into the codebook update
4. If you cannot agree, escalate to the lead researcher

**Do not average numeric fields — always discuss and agree on a single value.**

## Inter-rater reliability (IRR)

After reconciliation, run the IRR script:

```bash
python scripts/step11_irr_analysis.py \
  --h1 documentation/coding/systematic-map/rounds/FT-R1a/coding_H1.csv \
  --h2 documentation/coding/systematic-map/rounds/FT-R1a/coding_H2.csv \
  --gold documentation/coding/systematic-map/rounds/FT-R1a/reconciled.csv \
  --llm documentation/coding/systematic-map/rounds/FT-R1a/coding_LLM.csv \
  --out documentation/coding/systematic-map/rounds/FT-R1a/irr_R1a.json
```

This produces kappa and sensitivity per field and overall.

### Interpreting kappa (Cohen's κ)

| κ | Interpretation | Action |
|---|---|---|
| < 0.20 | Slight | Major codebook revision needed — field may need redesign |
| 0.21–0.40 | Fair | Significant revision — add more examples, tighten definitions |
| 0.41–0.60 | Moderate | Revision needed — review specific disagreement patterns |
| 0.61–0.80 | **Substantial** | Approaching threshold — fine-tuning only |
| > 0.80 | Almost perfect | Field is ready to lock |

**Minimum threshold to lock a closed field: κ ≥ 0.60 AND sensitivity ≥ 0.80** (in the same batch).

### Sensitivity for closed fields

For fields with a "positive" class (e.g. `cost_data_reported = yes`), sensitivity = TP / (TP + FN). This matters because missing a positive case is worse than a false positive — analogous to the abstract screening stage.

### Emergent fields (adaptation_focus, indicators_measured)

Emergent fields cannot be assessed with kappa in the same way. Instead, track:
- **Category saturation:** are new codes appearing in this batch that did not appear in the previous one?
- **Overlap rate:** what proportion of codes from H1 appear (verbatim or near-match) in H2?

Emergent fields are saturated when no substantively new codes appear across two consecutive batches.

---

## Stopping rules

### Closed fields (all except `adaptation_focus` and `indicators_measured`)
A field is **locked** once it reaches κ ≥ 0.60 AND sensitivity ≥ 0.80 in the same batch. Once locked, that field is not revised in subsequent rounds.

### Emergent fields (`adaptation_focus`, `indicators_measured`)
These fields are **saturated** when no new codes appear across two consecutive batches. At saturation, group the free-text codes into a controlled vocabulary and lock.

### Full corpus run
Proceed to the full corpus LLM run (step15) once:
- All closed fields are locked, AND
- Both emergent fields are saturated

Expected convergence: 3–4 rounds (300–400 papers). Stop earlier if criteria are met.

## LLM criteria tracking

Analogous to the abstract screening stage, the LLM extraction guidance is versioned and updated after each round. This ensures we can trace exactly what prompt the LLM was running in each batch and why it changed.

### Folder

```
documentation/coding/systematic-map/llm-criteria/
├── CHANGELOG.md        ← one-line entry per version: round, date, summary of changes
├── criteria_v1.md      ← initial LLM prompt + field-level extraction instructions (used in FT-R1a)
├── criteria_v2.md      ← updated after FT-R1a reconciliation (used in FT-R1b or R2a)
└── ...
```

### When and how to update

After **Step 4** (review disagreements) of each round:

1. Identify fields where the LLM underperformed (κ < 0.60 or sensitivity < 0.80)
2. For each underperforming field, determine the root cause:
   - Ambiguous instruction → rewrite the field definition in the criteria file
   - Missing example → add an example to the criteria file
   - Category gap (emergent field) → add the new code to the criteria file
3. Copy `criteria_vN.md` → `criteria_v(N+1).md` and apply the changes
4. Add one line to `CHANGELOG.md`: version, round, date, what changed and why
5. Run `step15` in FT mode using the new criteria file for the next batch

### What the criteria file contains

Each `criteria_vN.md` includes:
- **Version metadata:** which version, which rounds used it, date created
- **Task description:** what the LLM is asked to do
- **Field-by-field extraction instructions:** more explicit than the human guide — tells the LLM exactly where to look and how to decide
- **Decision rules for edge cases:** specific ambiguous cases that came up in reconciliation
- **Change log entry:** what changed from the previous version and why

The initial version is at [`llm-criteria/criteria_v1.yml`](llm-criteria/criteria_v1.yml) (loaded by step15)
with a human-readable companion at [`llm-criteria/criteria_v1.md`](llm-criteria/criteria_v1.md).
